# 参考项目

## 优先级说明

| 优先级 | 含义 |
|--------|------|
| **P0** | 核心架构参考，直接影响 IR 设计和编解码器接口 |
| **P1** | 重要参考，影响协议转换规则和路由设计 |
| **P2** | 辅助参考，提供生态视野和扩展方向 |

## P0 参考

### llmapimux (Go)

- **参考内容**：2N codec 架构、IR 类型定义、SSE 流处理
- **关键思路**：
  - 每个协议实现一对 Codec（Decode → Encode），N 个协议 = 2N 个转换器，而非 N² 个适配器
  - 统一 IR 作为中间表示层，解耦协议解析与生成
  - SSE 流以逐事件方式处理，避免缓冲完整响应
- **借鉴到本项目**：`Codec` trait 已采纳此设计（`decode_request` + `encode_response`），✅ 已落实

### Higress (Go/C++)

- **参考内容**：WASM plugin 接口设计 → 简化为 Rust trait 设计
- **关键思路**：
  - 通过可插拔行为（路由、认证、转换）扩展网关能力
  - 每个行为独立封装，通过接口组合
- **借鉴到本项目**：`Router`、`Converter`、`Authenticator` 三个 trait 已替代 WASM 插件机制，✅ 已落实

## P1 参考

### CCRelay (TS / TypeScript)

- **参考内容**：跨协议转换规则、hostname-specific alignment
- **关键思路**：
  - 基于模型名称或 API 端点标识目标协议，自动进行格式转换
  - 字段映射表驱动，非硬编码
  - 处理不同厂商对同一概念的不同命名（如 `metadata` vs `user`）
- **待借鉴**：`ProtocolConversion` 实体映射表的设计细节，stop_reason 枚举的全量映射

### copilot-api (TS / TypeScript)

- **参考内容**：多协议路由、provider model mapping
- **关键思路**：
  - 模型名称 → 后端提供商 + 协议的映射关系
  - 支持同一后端暴露多种协议接口
  - 请求特征检测（是否有 tools、是否 stream）影响路由决策
- **待借鉴**：`RouteRule` 的细化设计，模型别名映射机制

## P2 参考

### litellm (Python)

- **参考内容**：100+ provider 适配器列表、统一格式映射
- **关键思路**：
  - 大规模多提供商适配，覆盖 100+ 厂商
  - 统一成本追踪和日志格式
  - 配置驱动 vs 代码驱动的权衡
- **待借鉴**：长远扩展的适配器注册机制，日志和计费接口预留

### new-api (Go)

- **参考内容**：多协议聚合、用户管理
- **关键思路**：
  - 多渠道聚合，统一用户管理和计费
  - 渠道级别的负载均衡和故障转移
- **待借鉴**：`Router::on_error` 的 fallback 路由链设计

## 架构决策记录

所有以上参考项目均采用 **中继/代理** 模式（接收请求 → 转换 → 转发 → 转换 → 返回），
与 LLM Mux 定位一致。核心差异在于：

1. **2N vs N²**：llmapimux 的 2N codec 设计避免了协议数量的组合爆炸，是本项目的核心架构理念
2. **Trait vs Plugin**：用 Rust trait 替代 WASM/动态插件，获得编译期类型安全和零运行时开销
3. **无状态**：不实现用户管理、计费等业务功能，聚焦协议互转这一单一职责
